# 운영자 문의 작업공간

## 목적

운영자 작업공간은 제품의 핵심 작업 영역이다.

운영자는 이 화면에서 아래 일을 할 수 있어야 한다.

- 문의를 찾고,
- 맥락을 이해하고,
- 고객에게 응답하고,
- 내부 메모를 남기고,
- 케이스 상태를 관리한다.

## 주요 화면

### 문의 목록 화면

이 화면은 운영자의 triage 화면이다.

포함해야 하는 요소:

- 검색/필터 영역,
- KPI 요약 영역,
- 문의 목록,
- 눈에 보이는 상태와 유형,
- 고객 이름,
- 연결된 주문/상품 요약,
- 최근 메시지 시각,
- 최근 응답 시각,
- 연결된 챗봇 표시.

### 문의 상세 화면

이 화면은 운영자의 실제 처리 화면이다.

포함해야 하는 요소:

- 문의 헤더,
- 고객 요약,
- 주문/상품 요약,
- 문의 타임라인,
- 연결된 챗봇 맥락,
- 요약 및 안전 보조 영역,
- 내부 메모,
- 응답 입력,
- 상태 변경 액션.

## 주요 운영자 흐름

1. 운영자가 문의 목록을 연다.
2. 운영자가 필터 또는 검색을 조정한다.
3. 운영자가 특정 문의를 선택한다.
4. 운영자가 고객/주문/상품/문의/챗봇 맥락을 검토한다.
5. 운영자가 응답을 작성한다.
6. 운영자가 상태를 갱신한다.
7. 필요하면 내부 메모를 추가한다.

## 내부 메모

- 내부 메모는 운영자에게만 보인다.
- 고객과 주고받는 메시지와 분리된다.
- 문의와 함께 저장된다.
- 메모 이력 UI는 단순해도 된다.

## 상태 처리

운영자는 문의를 주요 상태 흐름에 따라 이동시킬 수 있어야 한다.

최소 지원 동작은 아래와 같다.

- 문의 접수,
- 처리 진행,
- 필요 시 대기/보류,
- 완료 처리,
- 완료 후 고객 메시지가 오면 재오픈.

여기서 재오픈은 새로운 별도 상태를 만드는 것이 아니라, 문의를 다시 `OPEN` 상태로 되돌리는 액션으로 본다.

상세한 conflict 처리나 reason code 체계는 이 명세에서 요구하지 않는다.

## 단순화 메모

이 문서는 아래를 완전하게 정의하지 않는다.

- 담당자 충돌 규칙,
- 동시 처리 규칙,
- 깊은 degraded 상태 매트릭스,
- 운영자/고객 분리 외의 세분 권한 분기,
- 이벤트 로그 중심 복구 동작.
