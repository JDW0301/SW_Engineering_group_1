# 기존 문서 누락·미비 사항 분석

## 1. 분석 목적

본 문서는 기존 `project-handbook` 문서를 기준으로, 구현 착수에 필요한 수준까지 내려갔을 때 무엇이 비어 있는지 진단하기 위한 문서이다.

## 2. 대분류 기준

기존 문서에서 추출한 대분류는 다음과 같다.

1. 제품 범위 및 MVP 경계
2. 도메인 모델 및 데이터 구조
3. 계정, 인증, 권한
4. 문의/상담 워크플로우
5. 스토어·상품·주문·고객 운영
6. 채팅 및 첨부 파일
7. AI 보조 기능 및 지식 자산
8. 검색 및 운영 지표
9. 외부 연동 및 동기화
10. 보안, 개인정보, 보관 정책
11. 예외 처리 및 공통 제약
12. 모듈 책임 및 운영 책임

## 3. 대분류별 분석

## 3.1 제품 범위 및 MVP 경계

### 현재 정의된 내용

- 자사몰용 통합 CS 플랫폼이라는 제품 방향
- 기본 운영 기능, 채팅, AI 기능, 보안 요구사항, 고도화 방향
- 단계별 우선순위 제안

### 비어 있는 부분

- MVP에 반드시 포함되는 기능과 제외되는 기능의 확정 기준
- 웹/앱 범위 확정
- 사업자 전용 도구와 소비자 화면의 포함 범위

### 추가로 정의해야 하는 부분

- 릴리스 기준 범위표
- 기능별 `필수/선택/보류` 구분
- MVP 완료 조건

### 선행 정의 필요 여부

**예.** 범위가 고정되어야 이후 문서가 과도하게 확장되지 않는다.

## 3.2 도메인 모델 및 데이터 구조

### 현재 정의된 내용

- 스토어, 상품, 고객, 주문, 문의, 채팅, FAQ, 정책, 프리셋 등 핵심 엔티티 명칭
- 고객별 주문/문의 이력 연결 필요성

### 비어 있는 부분

- 각 엔티티의 필수 필드와 선택 필드
- ID 규칙, 상태값, 관계 cardinality
- 원본 데이터와 편집 데이터의 출처 및 소유권

### 추가로 정의해야 하는 부분

- 데이터 사전
- 엔티티 관계도
- 공통 명명 규칙

### 선행 정의 필요 여부

**예.** 데이터 구조가 고정되어야 API, 화면, 검색, 통계, AI 입력 구조를 정할 수 있다.

## 3.3 계정, 인증, 권한

### 현재 정의된 내용

- 사업자 관리자 계정과 고객 계정 구분
- 사용자 유형별 접근 제한 필요성

### 비어 있는 부분

- 실제 로그인 방식
- 역할 계층 구조
- 기능별 권한 범위
- 권한 실패 시 처리 기준

### 추가로 정의해야 하는 부분

- 역할 정의표
- 권한 매트릭스
- 계정 생명주기 규칙

### 선행 정의 필요 여부

**예.** 데이터 노출 범위와 기능 접근 정책의 기준이 된다.

## 3.4 문의/상담 워크플로우

### 현재 정의된 내용

- 문의 접수, 상태 관리, 분류, 목록, 메모, 상담 기록
- 상태 예시: 접수, 대기, 진행 중, 종료, 보류

### 비어 있는 부분

- 상태 전이 규칙
- 담당자 배정 방식
- 재오픈, 중복 문의, AI 실패, 미응답 장기 방치 처리

### 추가로 정의해야 하는 부분

- 상태 머신
- 상태별 필수 입력값
- 전환 권한과 예외 처리

### 선행 정의 필요 여부

**예.** 화면 구성, 알림, 통계, AI handoff 규칙이 모두 여기에 의존한다.

## 3.5 스토어·상품·주문·고객 운영

### 현재 정의된 내용

- 통합 조회 필요성
- 스토어/상품/고객/주문 등록·조회 또는 관리 기능 필요성

### 비어 있는 부분

- 각 엔티티가 읽기 전용인지 수정 가능한지
- 수동 입력과 외부 동기화 충돌 시 우선순위
- 일부 데이터가 없을 때 조회 화면 동작

### 추가로 정의해야 하는 부분

- 엔티티별 CRUD 범위
- 동기화 우선권 규칙
- 조회 fallback 규칙

### 선행 정의 필요 여부

**예.** 통합 상담 화면과 문의 처리 정확도의 기반이 된다.

## 3.6 채팅 및 첨부 파일

### 현재 정의된 내용

- 실시간 채팅, 채팅방, 이력 저장, 이미지/영상 첨부, 읽음 여부, 마지막 응답 시간

### 비어 있는 부분

- 채팅방 생성 시점
- 메시지 상태 모델
- 허용 파일 형식, 크기 제한, 보관 기간
- 업로드 실패 및 미리보기 실패 처리

### 추가로 정의해야 하는 부분

- 메시지 단위 스키마
- 첨부 파일 정책
- 채팅 이벤트 흐름

### 선행 정의 필요 여부

**중간.** 핵심 워크플로우 이후, 실시간 기능 구현 전에 정의되어야 한다.

## 3.7 AI 보조 기능 및 지식 자산

### 현재 정의된 내용

- 맞춤형 챗봇, 메시지 요약, 악성 표현 감지, 중립적 표현 변환, FAQ/정책/금지 답변 규칙

### 비어 있는 부분

- AI 기능별 입력/출력 계약
- 문서 우선순위와 충돌 해결 규칙
- AI 실패 기준
- 운영자가 수정 가능한 지식 자산의 단위

### 추가로 정의해야 하는 부분

- AI 기능별 정책 문서
- 지식 자산 버전 규칙
- 인간 상담 전환 기준

### 선행 정의 필요 여부

**중간 이상.** 기본 문의 처리 흐름 정의 후, AI 기능 착수 전에 반드시 필요하다.

## 3.8 검색 및 운영 지표

### 현재 정의된 내용

- 검색 기준: 주문번호, 고객명, 상품명, 문의유형, 기간
- 지표 예시: 일/주/월 문의 수, 평균 응답 시간, 해결 완료율

### 비어 있는 부분

- 부분 일치/정확 일치 기준
- 정렬 우선순위
- 응답 시간 계산 기준
- 재오픈 문의 처리 기준

### 추가로 정의해야 하는 부분

- 검색 계약서
- 지표 계산 정의서

### 선행 정의 필요 여부

**아니오.** 도메인 모델과 상태 모델이 먼저 고정되어야 한다.

## 3.9 외부 연동 및 동기화

### 현재 정의된 내용

- API 또는 웹훅 기반 연동 가능성
- 주문/상품 동기화 필요성

### 비어 있는 부분

- 연동 대상 시스템
- 동기화 방식별 입력 포맷
- 실패 재시도, 수동 재동기화, 중복 이벤트 처리

### 추가로 정의해야 하는 부분

- 연동 경계 정의서
- 동기화 정책
- 장애 복구 규칙

### 선행 정의 필요 여부

**조건부 예.** 외부 데이터 유입을 실제 구현할 경우 선행되어야 한다.

## 3.10 보안, 개인정보, 보관 정책

### 현재 정의된 내용

- 접근 제한, 스토어 단위 접근 제한, 민감정보 마스킹, 보관 정책 필요성

### 비어 있는 부분

- 민감정보 필드 목록
- 역할별 마스킹 규칙
- 삭제와 익명화 기준
- 감사 로그 범위

### 추가로 정의해야 하는 부분

- 데이터 분류표
- 보관/파기 정책
- 접근 로그 정책

### 선행 정의 필요 여부

**예.** API, 파일 저장, AI 처리 범위를 모두 제한하는 기준이 된다.

## 3.11 예외 처리 및 공통 제약

### 현재 정의된 내용

- AI 응답 실패, 동기화 실패 가능성 언급

### 비어 있는 부분

- 잘못된 입력 처리
- 권한 없는 접근 처리
- 첨부 파일 오류 처리
- 누락된 연결 데이터 처리
- 시간 초과, 재시도, 사용자 표시 문구 기준

### 추가로 정의해야 하는 부분

- 예외 상황 목록
- 오류 코드/오류 유형 체계
- 사용자 노출 메시지 기준

### 선행 정의 필요 여부

**예.** 상세 기능 문서를 실제 구현 수준으로 낮추기 위한 공통 기반이다.

## 3.12 모듈 책임 및 운영 책임

### 현재 정의된 내용

- FAQ, 정책 문서, 챗봇 기준 문서를 운영자가 관리할 수 있다는 방향성

### 비어 있는 부분

- 어떤 모듈을 누가 수정하고 승인하는지
- 정책 충돌 시 최종 결정 주체
- AI 예외 상황 대응 책임자

### 추가로 정의해야 하는 부분

- 모듈 책임 매트릭스
- 승인 주체 및 운영 책임 규칙

### 선행 정의 필요 여부

**중간.** 구현 시작 직후보다 운영 설계 단계에서 중요하지만, 관리자 기능 설계에는 미리 반영되어야 한다.

## 4. 요약 결론

가장 큰 공백은 다음 여섯 가지이다.

1. MVP 범위 경계
2. 도메인 모델 및 데이터 사전
3. 권한 모델
4. 문의 상태 머신
5. 개인정보/보관/마스킹 정책
6. 예외 처리 및 공통 제약

이 여섯 항목이 먼저 고정되지 않으면, 나머지 기능 명세는 표현만 자세할 뿐 실제 구현 기준으로는 불완전한 상태에 머문다.
