# 역할 및 권한 모델

## 1. 목적

본 문서는 계정 유형, 역할별 책임, 데이터 접근 범위, 기능 수행 권한을 정의한다.

## 2. 역할 정의

## 2.1 merchant_operator

- 스토어 고객센터 운영 주체
- 문의 목록, 문의 상세, 상태 변경, 내부 메모, 프리셋/FAQ/정책 관리, 챗봇 세션 조회, 수동 재동기화 수행 가능

## 2.2 customer

- 로그인한 회원 고객
- 본인 문의 등록 및 조회, 본인 챗봇 세션 이용, 본인 대화 원문 확인 주체

관리자/상담자 역할 세분화는 MVP 후속 확장으로 둔다.

## 3. 권한 범위 기준

## 3.1 데이터 스코프 규칙

- 운영자는 자신이 소속된 `store_id` 범위 데이터에만 접근 가능하다.
- 고객은 자신에게 귀속된 문의, 챗봇 세션, 메시지, 첨부 파일에만 접근 가능하다.
- 내부 메모와 내부 운영 데이터는 고객에게 노출되지 않는다.

## 3.2 권한 결정 순서

권한 판단은 아래 순서로 수행한다.

1. 인증 여부 확인
2. 역할 확인
3. 스토어 범위 확인
4. 리소스 소유 관계 확인
5. 행위 권한 확인 (조회/생성/수정/상태 변경/재동기화)

## 4. 권한 매트릭스

| 기능 | merchant_operator | customer |
|---|---|---|
| 문의 목록 조회 | 가능 | 본인 문의만 가능 |
| 문의 상세 조회 | 가능 | 본인 문의만 가능 |
| 문의 상태 변경 | 가능 | 불가 |
| 내부 메모 작성 | 가능 | 불가 |
| FAQ/정책/프리셋 관리 | 가능 | 불가 |
| 챗봇 지식 파일 업로드 | 가능 | 불가 |
| 챗봇 세션 조회 | 가능 | 본인 세션만 가능 |
| 채팅/문의 메시지 발신 | 가능 | 본인 문의/세션만 가능 |
| 첨부 파일 업로드 | 가능 | 허용 여부 추가 정의 필요 |
| 고객 정보 조회 | 가능 | 본인 정보 일부만 가능 |
| 외부 데이터 수동 재동기화 | 가능 | 불가 |
| 운영 지표 조회 | 가능 | 불가 |

## 5. 인증 및 세션 규칙 초안

- 고객은 회원 로그인 기반 인증을 사용한다.
- 운영자는 운영자 앱 로그인 기반 인증을 사용한다.
- MVP의 기본 인증 방식은 세션 기반 인증이다.
- 세션은 브라우저 종료 시 만료한다.
- MVP에서는 별도의 비활성 타임아웃을 두지 않는다.
- 세션 만료 후 재접속 시 로그인 페이지로 유도한다.
- 비밀번호 정책의 세부 기준은 **추가 정의 필요**하다.

## 6. 권한 실패 처리 규칙

- 인증되지 않은 사용자는 로그인 또는 인증 페이지로 유도한다.
- 인증되었으나 권한이 없는 경우, 리소스 존재 여부를 노출하지 않는 방식이 바람직하다.
- 고객이 다른 고객의 문의 또는 챗봇 세션에 접근하려는 경우 `권한 없음`으로 처리한다.
- 내부 메모, 내부 운영 설정, 내부 처리 로그는 고객에게 절대 노출되지 않는다.

## 7. 원문 보기 및 마스킹 연계 규칙

- 기본 UI/API 출력은 `content_display` 기준으로 한다.
- 운영자는 고객 공개 대상 대화 영역의 원문(`content_raw`)을 확인할 수 있다.
- 고객도 본인이 참여하는 고객 공개 대상 대화 영역의 원문을 확인할 수 있다.
- 아래 항목은 원문 보기 대상에서 제외한다.
  - 내부 메모
  - 내부 운영 판단 데이터
  - 시스템 내부 처리용 중간 데이터

## 8. 모듈 책임 연계

- 권한 검사는 UI 단과 API 단 모두에서 수행해야 한다.
- 화면에서 숨겨진 버튼만으로 권한이 보장되었다고 보지 않는다.
- 데이터 조회 API는 반드시 스코프 검사를 수행해야 한다.

## 9. 추가 정의 필요 사항

- 운영자 초대/생성 방식
- 계정 비활성화와 탈퇴 차이
- 스토어 다중 소속 사용자 허용 여부
- 원문 보기 버튼의 구체적 노출 조건
