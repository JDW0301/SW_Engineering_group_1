# 고객 앱 페이지

## 목적

이 문서는 고객이 어떤 순서로 지원 기능을 사용하는지 기준으로 필요한 페이지를 정리한다.

핵심은 고객이 복잡한 절차 없이 스토어를 찾고, 필요하면 주문을 연결하고, `문의` 또는 `챗봇`을 통해 문제를 해결하도록 만드는 것이다.

## 고객 앱 페이지 목록

고객 앱의 기본 페이지는 아래 여섯 가지로 본다.

1. 고객 지원 진입 페이지
2. 내 스토어 목록/검색 페이지
3. 주문 연결(claim/link) 페이지
4. 고객 문의 목록 페이지
5. 고객 문의 상세 페이지
6. 챗봇 세션 페이지

## 1. 고객 지원 진입 페이지

### 목적

로그인한 고객이 지원 기능 전체로 들어오는 가장 상위 진입 화면이다.

### 포함 기능

- `문의` 탭
- `챗봇` 탭
- 현재 선택된 스토어 표시
- 스토어를 아직 고르지 않았을 때 스토어 선택 유도
- 문의 목록 또는 챗봇 세션으로 이어지는 상위 내비게이션

### 주요 흐름

1. 고객이 로그인 후 지원 영역으로 들어온다.
2. 고객이 먼저 스토어를 고른다.
3. 고객이 `문의` 또는 `챗봇` 탭을 선택한다.

### 단순화 메모

- 이 화면은 실제 세부 기능을 모두 담는 대형 대시보드가 아니다.
- 상위 탭과 진입 분기만 명확하면 충분하다.

## 2. 내 스토어 목록/검색 페이지

### 목적

고객이 자신과 연결된 스토어를 빠르게 고르거나, 새 스토어를 검색해 지원 맥락을 잡는 페이지다.

### 포함 기능

- 기존 연결 스토어 목록
- 최근 구매 또는 최근 문의가 있었던 스토어 요약
- 스토어 검색 입력
- 검색 결과 표시
- 스토어 선택 액션

### 주요 흐름

1. 고객이 지원 진입 후 스토어 선택이 필요하다는 안내를 본다.
2. 내 스토어 목록에서 바로 고르거나 검색을 사용한다.
3. 스토어를 선택한 뒤 문의 또는 챗봇 흐름으로 이동한다.

### 단순화 메모

- 신규 가입 고객은 이 목록이 비어 있을 수 있다.
- 이 경우 검색 기반 선택을 기본 흐름으로 둔다.

## 3. 주문 연결(claim/link) 페이지

### 목적

고객이 외부 주문 이력을 자신의 로컬 계정과 연결하는 페이지다.

### 포함 기능

- 주문번호 입력
- 기본 고객 정보 힌트 입력
- 연결 후보 주문 표시
- claim/link 요청 액션
- 필요 시 mock verification 진행 액션
- 현재 연결 상태 표시

### 주요 흐름

1. 고객이 특정 스토어 안에서 주문 연결이 필요하다는 안내를 본다.
2. 주문번호와 기본 힌트를 입력한다.
3. 연결 후보를 확인한다.
4. 필요하면 mock verification을 수행한다.
5. 성공하면 주문 snapshot이 고객 맥락에 연결된다.

### 단순화 메모

- claim이 끝나지 않아도 문의 생성 자체는 가능하게 둔다.
- 다만 주문 연결 상태는 미확정으로 표시할 수 있다.
- mock verification은 실제 인증 연동이 아닌 학교 프로젝트용 시뮬레이션이다.

## 4. 고객 문의 목록 페이지

### 목적

고객이 본인의 문의를 한눈에 확인하고 어떤 문의를 다시 열지 결정하는 페이지다.

### 포함 기능

- 문의 목록 표시
- 고객이 볼 수 있는 상태 표시
- 최근 메시지 시각 표시
- 연결된 주문/상품 요약 표시
- 문의 없음 안내
- 새 문의 시작 액션
- 챗봇 이동 유도

### 주요 흐름

1. 고객이 `문의` 탭으로 진입한다.
2. 자신의 문의 목록을 본다.
3. 필요하면 같은 화면에서 `새 문의 시작` 액션을 눌러 새 문의 작성 흐름으로 들어간다.
4. 기존 문의가 있으면 특정 문의를 선택해 상세 페이지로 이동한다.

### 새 문의 시작 방식

이 프로젝트에서는 고객의 직접 문의 생성 경로를 별도 대형 페이지로 분리하지 않아도 된다.

기본 해석은 아래와 같다.

- 고객은 `문의` 탭 안의 문의 목록 화면에서 새 문의를 시작한다.
- 새 문의 시작 시 스토어 맥락은 이미 선택되어 있다고 본다.
- 필요하면 주문 연결 상태를 확인하거나 claim/link 화면으로 잠깐 이동할 수 있다.
- 문의 생성이 끝나면 바로 고객 문의 상세 페이지로 들어가 후속 메시지를 이어간다.

### 단순화 메모

- 고객은 자신의 문의만 볼 수 있다.
- 고객용 검색/고급 필터는 MVP 기본 범위에서 생략해도 된다.
- 새 문의 작성 폼은 문의 목록 화면 안의 버튼, 인라인 폼, 또는 가벼운 작성 단계로 구현하면 충분하다.

## 5. 고객 문의 상세 페이지

### 목적

고객이 현재 문의 상태와 대화 내용을 확인하고, 후속 메시지를 보내는 페이지다.

### 포함 기능

- 현재 상태 표시
- 문의 유형 또는 간단 요약
- 메시지 타임라인
- 고객 메시지 입력
- 첨부 업로드
- `content_display` 기준 메시지 표시
- 필요 시 raw 보기 액션
- 주문/상품 요약
- `[상세 보기]` 확장 영역

### 주요 흐름

1. 고객이 문의 목록에서 문의를 선택한다.
2. 상태와 이전 대화를 확인한다.
3. 새 메시지나 첨부를 보낸다.
4. 필요하면 주문/상품 세부를 펼쳐 본다.

### 단순화 메모

- 고객은 문의 상태를 직접 수정하지 않는다.
- 내부 메모와 운영자 전용 데이터는 노출하지 않는다.
- 주문/상품 정보가 일부 누락되어도 문의 대화 자체는 계속 볼 수 있어야 한다.

## 6. 챗봇 세션 페이지

### 목적

고객이 self-service 챗봇과 대화하고, 필요 시 관리자 응대로 handoff를 요청하는 페이지다.

### 포함 기능

- 세션 상태 표시
- 챗봇 대화 타임라인
- 고객 질문 입력
- 첨부 업로드
- 챗봇 응답 표시
- handoff 버튼
- 이미 연결된 문의가 있으면 해당 문의로 이동하는 액션
- 세션 만료 시 read-only 안내와 새 세션 시작 액션

### 주요 흐름

1. 고객이 `챗봇` 탭으로 진입한다.
2. 새 세션을 시작하거나 기존 세션을 연다.
3. 질문을 입력하고 답변을 본다.
4. self-service로 해결되지 않으면 handoff를 요청한다.
5. 시스템이 문의를 생성하거나 기존 연결 문의로 안내한다.
6. 이후 고객은 문의 상세 페이지에서 관리자와의 문의 대화를 이어간다.

### handoff 규칙

- handoff는 active chatbot session에서 언제든 요청할 수 있다.
- 같은 세션에서 handoff를 다시 눌러도 새 문의를 계속 만들지 않는다.
- 이미 연결된 문의가 있으면 그 문의로 보내는 것이 기본이다.
- `chatbot_session`은 `inquiry`와 같은 것이 아니라 별도 흐름이다.

### 단순화 메모

- 고급 세션 재시도 정책은 자세히 다루지 않는다.
- 중요한 점은 챗봇이 self-service 진입점이면서 관리자 응대 연결 통로라는 것이다.

## 고객 앱 전체 흐름 요약

1. 로그인 후 지원 진입 화면으로 들어온다.
2. 스토어를 선택한다.
3. 필요하면 주문 연결을 한다.
4. `문의` 탭에서는 직접 새 문의를 만들거나 기존 문의를 연다.
5. `챗봇` 탭에서는 self-service를 먼저 시도하고 필요하면 handoff를 요청한다.
6. handoff 이후에는 문의 상세를 중심으로 관리자 응대를 이어간다.
