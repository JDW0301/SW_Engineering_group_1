# 공용 진입과 라우팅 페이지

## 목적

이 문서는 고객 앱과 운영자 앱으로 갈라지기 전, 제품 전체에서 공통으로 이해해야 하는 진입 구조와 라우팅 원칙을 정리한다.

핵심은 두 앱을 하나로 섞지 않고, 같은 제품 안의 서로 다른 사용자 경험으로 분리하는 것이다.

## 제품 수준 진입 구조

이 프로젝트의 사용자 표면은 아래 두 개의 앱 엔트리로 분리한다.

- `customer-app`: 고객용 앱
- `operator-app`: 운영자용 앱

두 앱은 같은 백엔드와 같은 핵심 도메인을 공유하지만, 화면 흐름은 분리한다.

이 문서에서 `운영자`, `관리자`, `상담자`가 함께 보이더라도 다른 역할을 뜻하지 않는다. `운영자`는 앱 사용 문맥, `관리자`는 계정/권한/응대 책임 문맥, `상담자`는 고객 응대 행위 문맥에서 쓴 표현일 뿐이며, 모두 같은 사람을 가리킨다.

## 공용 진입 페이지

### 1. 앱 선택 / 로그인 진입 페이지

이 페이지는 사용자가 어떤 앱으로 들어갈지 결정하는 공용 진입점이다.

포함 기능:

- 고객 로그인 진입
- 운영자 로그인 진입
- 현재 서비스가 고객 지원 플랫폼이라는 간단한 안내

단순화 원칙:

- 고객과 운영자의 인증 구조를 깊게 나누지 않는다.
- 학교 프로젝트 수준에서는 앱 분기와 로그인 진입만 분명하면 된다.

### 2. 앱별 홈 진입 라우팅

로그인 후에는 역할에 따라 각 앱의 첫 페이지로 이동한다.

- 고객은 고객 지원 진입 화면으로 이동한다.
- 운영자는 문의 작업공간 목록 화면으로 이동한다.

이 문서는 공용 라우팅만 다루고, 이후 상세 행동은 각 앱 문서가 소유한다.

## 핵심 E2E 흐름

### 고객 중심 흐름

1. 고객이 로그인한다.
2. 고객이 고객 앱으로 진입한다.
3. 고객이 스토어를 선택한다.
4. 고객이 `문의` 또는 `챗봇` 중 하나를 선택한다.
5. 필요하면 챗봇에서 관리자 응대로 handoff 된다.
6. 생성되거나 연결된 문의에서 후속 대화를 이어간다.

### 운영자 중심 흐름

1. 운영자가 로그인한다.
2. 운영자가 운영자 앱으로 진입한다.
3. 운영자가 문의 목록에서 대상을 찾는다.
4. 문의 상세에서 맥락을 보고 응답한다.
5. 내부 메모, 상태 변경, 재오픈 등 후속 처리를 한다.

## 공용 라우팅 원칙

- `customer-app`과 `operator-app`의 경계를 넘는 이동은 명시적이어야 한다.
- 고객용 페이지 안에 운영자 전용 액션을 넣지 않는다.
- 운영자 페이지 안에 고객용 self-service 행동을 넣지 않는다.
- 챗봇에서 문의로 넘어가는 handoff는 앱 전환이 아니라 고객 앱 안의 흐름 전환으로 본다.

## 문의 중심 해석

이 프로젝트의 중심 엔티티는 `inquiry`다.

다만 고객은 챗봇을 먼저 사용할 수 있고, `chatbot_session`은 `inquiry`와 별도 흐름으로 존재한다. 따라서 공용 라우팅 관점에서는 아래를 기억하면 된다.

- 고객의 첫 self-service 진입점은 챗봇일 수 있다.
- 관리자 응대의 중심 화면은 결국 `inquiry`다.
- handoff는 챗봇 세션을 문의로 바꾸는 것이 아니라, 연결된 문의를 새로 만들거나 기존 문의로 연결하는 흐름이다.

## 단순화 메모

이 문서는 아래 항목을 자세히 정의하지 않는다.

- 세션 만료 세부 규칙
- 세부 권한 매트릭스
- 운영 로그 기반 복구 흐름
- 고급 로그인/인증 예외 처리
