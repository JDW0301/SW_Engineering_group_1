# 운영·보안·기술 메모

## 1. 운영 관점 메모

## 1.1 데이터 운영

- 서비스 구조에 맞는 DB 매뉴얼이 필요하다.
- 스토어, 상품, 고객, 주문, 문의 데이터 간 관계를 명확히 정의해야 한다.
- 고객별 주문 이력과 문의 이력이 연결되어야 상담 효율이 높아진다.

## 1.2 외부 연동 및 동기화

- 주문/상품 동기화 주기를 정의해야 한다.
- 동기화 방식으로 API 또는 웹훅 기반 연동이 언급되어 있다.
- 어떤 외부 서비스와 연동할 것인지는 **확인 필요**하다.
- 연동 실패 시 재시도 정책, 수동 동기화 여부는 **추가 구체화 필요**하다.

## 1.3 운영 매뉴얼

- DB 구조 설명 문서가 필요하다.
- 챗봇 답변 기준 문서와 정책 문서는 운영자가 수정 가능해야 한다.
- FAQ, 금지 답변, 예외 답변 규칙 등은 운영 지식 자산으로 관리할 필요가 있다.

## 2. 보안 요구사항

## 2.1 접근 통제

- 개인정보 접근 제한이 필요하다.
- 스토어 단위 접근 제한이 필요하다.
- 계정 유형에 따라 데이터 접근 범위를 제한해야 한다.

## 2.2 개인정보 및 민감정보 보호

- 민감정보는 적절히 마스킹해야 한다.
- 고객 데이터 열람 이력 또는 관리 정책은 **추가 검토 필요**하다.
- 개인정보 저장 범위와 최소 수집 원칙은 명확히 정의할 필요가 있다.

## 2.3 보관 및 삭제 정책

- 탈퇴 정책이 필요하다.
- 데이터 보관기간 정책이 필요하다.
- 관련 법규 반영 여부는 원문 기준으로 확인되지 않으므로 **확인 필요**하다.

## 3. 기술 스택 메모

원문에 명시된 기술 스택은 다음과 같다.

- DB: MySQL
- 프로그래밍 언어: Python
- 웹/앱 구현: ReAct

## 3.1 기술 표기 관련 확인 사항

- `ReAct` 표기는 일반적인 프론트엔드 프레임워크 명칭과 다를 수 있으므로 **React를 의미하는지 확인 필요**하다.
- 웹과 앱을 모두 구현 대상으로 볼지, 우선 웹만 구현할지는 별도 결정이 필요하다.

## 4. 구현 범위 메모

원문에는 다음 구현 범위가 언급되어 있다.

- DB
- 백엔드
- 프론트엔드
- 시간이 허용되면 웹/앱 서비스까지 고려

다만 이 표현은 초안 수준으로 보이며, 다음 항목은 명확히 다시 정의해야 한다.

- MVP 기준 구현 범위
- 관리자 웹과 소비자 화면의 분리 여부
- 앱 개발 포함 여부
- AI 기능의 실제 도입 범위

## 5. 운영 정책 측면에서 추가 정의가 필요한 항목

- 파일 저장 위치 및 저장소 정책
- 업로드 가능 파일 형식
- 파일 최대 크기
- 채팅 메시지 및 첨부 파일 보관 기간
- 문의 상태별 알림 정책
- 상담자 배정 방식

## 6. 기술 설계 전 선행 정리 권장사항

구현 착수 전 아래 항목을 먼저 문서화하는 것이 바람직하다.

1. 도메인 모델 정의
   - 스토어, 고객, 주문, 상품, 문의, 채팅, FAQ, 정책, 프리셋의 관계 정리
2. 권한 모델 정의
   - 사업자 관리자, 상담자, 고객의 권한 범위 정리
3. AI 운영 정책 정의
   - 답변 허용 범위, 금지 응답 규칙, 상담자 전환 기준 정리
4. 개인정보 처리 정책 정의
   - 저장 항목, 보관 기간, 마스킹 기준 정리

## 7. 현재 문서 기준 제한 사항

- 인프라 구조, 배포 환경, 개발 환경 구성은 원문에 포함되어 있지 않다.
- 테스트 전략, 품질 기준, 모니터링 방식은 아직 정의되지 않았다.
- AI 모델 종류, 호출 방식, 비용 통제 전략은 명시되어 있지 않다.
