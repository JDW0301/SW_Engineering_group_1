# 고객 포털

## 목적

고객 포털은 로그인한 고객이 지원팀과 소통하고 진행 중인 이슈를 추적할 수 있게 한다.

## 주요 화면

### 고객 지원 진입 화면

최상위 지원 진입 화면은 두 개의 탭을 제공해야 한다.

- `문의`
- `챗봇`

또한 고객이 문의를 시작하기 전에 스토어를 선택할 수 있어야 한다.

### 내 스토어 목록 및 검색 화면

이 화면은 고객이 이미 구매 이력이나 기존 연결 이력이 있는 스토어를 빠르게 고를 수 있게 한다.

신규 가입 직후에는 이 목록이 비어 있을 수 있으며, 이 경우 스토어 검색이 기본 진입 경로가 된다.

표시해야 하는 것:

- 내 스토어 목록,
- 최근 문의 또는 최근 구매가 있었던 스토어 요약,
- 새로운 스토어를 찾기 위한 검색 입력,
- 검색 결과에서 스토어 선택 액션.

여기서 말하는 내 스토어는 고객이 소유한 스토어가 아니라, 고객이 구매 또는 문의 이력을 가진 스토어 목록을 의미한다.

### 주문 연결 / claim 화면

신규 가입 고객이 기존 외부 주문을 자신의 계정과 연결할 때 사용하는 화면이다.

표시해야 하는 것:

- 주문번호 입력,
- 기본 고객 정보 힌트 입력,
- 연결 후보 주문 표시,
- claim 요청 액션,
- 필요 시 mock verification 진행 액션.

이 화면은 문의 생성 이전이나 문의 작성 중간 단계에서 호출될 수 있다.

### 고객 문의 목록 화면

고객 본인의 문의만 보여준다.

각 항목은 아래 정보를 보여줘야 한다.

- 문의 식별자 또는 요약,
- 고객이 볼 수 있는 상태,
- 최근 메시지 시각,
- 가능하다면 연결된 주문/상품 요약.

### 고객 문의 상세 화면

아래 내용을 보여준다.

- 현재 상태,
- 문의 메시지,
- 고객 메시지 입력,
- 첨부파일,
- 주문/상품 요약,
- 선택적 상세 확장 영역.

## 주요 고객 흐름

1. 고객이 로그인한다.
2. 고객이 로그인 완료 후 지원 영역으로 진입한다.
3. 고객이 내 스토어를 고르거나 스토어 검색으로 새 스토어를 찾는다.
4. 필요하면 기존 주문을 claim/link 한다.
5. 고객이 새 문의를 만들거나 기존 문의를 연다.
6. 고객이 메시지와 첨부를 보낸다.
7. 고객이 문의 상태를 확인한다.
8. 필요하면 대화를 계속 이어간다.

## 노출 규칙

- 고객은 자신의 문의만 볼 수 있다.
- 고객은 문의 상태를 직접 수정할 수 없다.
- 고객은 고객에게 공개되는 대화 내용을 볼 수 있다.
- 고객은 내부 메모나 운영자 전용 데이터를 볼 수 없다.

## 주문 연결과 검증 흐름

- 고객은 신규 가입 후 기존 외부 주문을 자신의 계정에 연결할 수 있다.
- 주문 연결은 로컬 계정과 외부 order/customer snapshot의 claim/link 흐름으로 본다.
- 기본 연결은 주문번호와 낮은 민감도의 고객 정보 힌트로 시도한다.
- 기본 연결만으로 충분하지 않은 경우에만 mock verification 같은 보조 확인 절차를 사용할 수 있다.
- mock verification은 실제 통신사 연동이 아니라 학교 프로젝트용 시뮬레이션 구현이다.
- 검증이 끝나지 않아도 문의 자체는 생성할 수 있지만, `order_snapshot` 연결 상태는 미확정으로 남을 수 있다.
- claim이 성공하기 전에는 문의가 스토어 기준으로만 생성될 수 있고, 주문 연결 시도 상태는 별도로 표시한다.

## 단순화 메모

이 명세는 고객 흐름을 의도적으로 단순하게 유지한다.

아래 항목은 완전하게 정의하지 않는다.

- 비회원 문의 지원,
- 다중 계정 소유권의 예외 케이스,
- 상세한 forbidden/not-found 분기,
- 고급 고객 권한 세분화.
