# 문의 상태 모델 및 처리 규칙

## 1. 목적

본 문서는 문의(`inquiry`)의 상태, 전환 규칙, 입력 조건, 예외 상황을 정의한다.

## 2. 상태 정의

| 상태값 | 한국어 의미 | 설명 |
|---|---|---|
| `OPEN` | 접수 | 문의가 생성되었고 아직 본격 처리 전인 상태 |
| `WAITING` | 대기 | 처리 대상이지만 즉시 응답 중이 아닌 상태 |
| `IN_PROGRESS` | 진행 중 | 운영자가 현재 대응 중인 상태 |
| `ON_HOLD` | 보류 | 추가 정보 부족, 외부 확인 필요 등으로 일시 정지된 상태 |
| `RESOLVED` | 종료 | 처리 완료로 간주되는 상태 |

위 다섯 상태값을 MVP 기준 상태 enum으로 확정한다.

## 3. 기본 전환 규칙

```text
OPEN -> WAITING
OPEN -> IN_PROGRESS
WAITING -> IN_PROGRESS
IN_PROGRESS -> ON_HOLD
IN_PROGRESS -> RESOLVED
ON_HOLD -> IN_PROGRESS
RESOLVED -> IN_PROGRESS  (재오픈)
```

## 4. 상태별 의미와 필수 조건

## 4.1 OPEN

- 문의가 최초 생성된 상태
- 필수 데이터: `inquiry_type`, 생성 시각
- 챗봇 self-service에서 즉시 상담 전환된 문의는 기본적으로 `OPEN`으로 생성한다.

## 4.2 WAITING

- 처리 큐에 들어가 있으나 적극 처리 중은 아닌 상태
- `assigned_operator_id` 없이도 가능하다.

## 4.3 IN_PROGRESS

- 운영자가 현재 대응 중인 상태
- 최소한 하나 이상의 운영자 응답 또는 명시적 처리 개시가 존재해야 한다.

## 4.4 ON_HOLD

- 추가 고객 정보, 배송 확인, 환불 검토 등 외부 조건을 기다리는 상태
- 보류 사유 기록을 권장한다.

## 4.5 RESOLVED

- 문의를 종료한 상태
- 종료 사유 또는 처리 결과 메모가 필요하다.
- 종료 후 재오픈 가능 여부는 허용한다.

## 5. 상태 전환 권한

- `merchant_operator`: 모든 전환 가능
- `customer`: 상태 직접 변경 불가
- 챗봇/자동화 기능: 상태 직접 전환 불가. 문의 생성 또는 전환 제안만 가능

## 6. 재오픈 규칙

- `RESOLVED` 상태의 문의에 고객 신규 메시지가 도착하면 재오픈 후보가 된다.
- 재오픈 시 기본 상태는 `IN_PROGRESS`를 우선 기본값으로 한다.
- 기존 담당자 유지 여부는 **추가 정의 필요**하다.

## 7. 챗봇 전환 규칙

- 고객은 챗봇 세션에서 원하면 언제든 상담 전환 버튼을 사용할 수 있다.
- 상담 전환 버튼 클릭 시 `inquiry`를 즉시 생성한다.
- 생성된 문의에는 연결된 `chatbot_session` 참조와 전환 사실을 남긴다.
- 최근 챗봇 대화 또는 요약은 운영자가 확인할 수 있어야 한다.
- 유의미한 챗봇 기록이 없고 고객 추가 메시지도 없으면 문의를 자동 만료 대상으로 둘 수 있다.

## 8. 예외 상황 처리

## 8.1 중복 문의

- 동일 고객이 짧은 시간 내 유사 문의를 반복 등록할 수 있다.
- 기본 원칙은 별도 문의로 저장하되, 운영자가 연결 관계를 인지할 수 있어야 한다.

## 8.2 연결 데이터 누락

- 주문번호가 없거나 매칭 실패한 문의도 접수 가능해야 한다.
- 이 경우 `order_id`는 비어 있을 수 있으며, 후속 매핑이 가능해야 한다.

## 8.3 자동 만료 대상 챗봇 전환 문의

- 챗봇 전환으로 즉시 생성된 문의는 연결된 챗봇 기록이 실질적으로 비어 있고 고객 추가 메시지도 없는 경우에만 10분 후 자동 만료 대상으로 둘 수 있다.
- 일반 문의에는 자동 만료 규칙을 적용하지 않는다.

## 9. 시간 필드 처리 기준

- `opened_at`: 문의 생성 시각
- `last_message_at`: 가장 최근 메시지 도착 시각
- `last_response_at`: 운영자 측 마지막 응답 시각
- `closed_at`: 종료 처리 시각

응답 시간 계산은 이 시간 필드 정의를 기준으로 수행한다.

## 10. 모듈 책임

- 문의 목록 모듈: 상태 기반 정렬/필터링
- 문의 상세 모듈: 상태 전환 UI 및 이력 표시
- 통계 모듈: 상태와 시간 필드 기반 지표 계산
- 자동화 모듈: 상태 직접 확정이 아니라 문의 생성 또는 보조 판단 제공

## 11. 추가 정의 필요 사항

- 보류 사유 코드 체계
- 종료 사유 코드 체계
- 재오픈 시 담당자 유지 여부
