# 통합 상담 작업공간 및 문의 운영 명세

## 1. 목적

본 문서는 운영자가 실제로 사용하는 핵심 작업공간의 화면 책임, 입력/출력, 데이터 흐름, 예외 상황을 정의한다.

## 2. 모듈 구성

## 2.1 inquiry-list-module

### 책임

- 문의 목록 조회
- 상태/유형/기간 기반 필터링
- 정렬 기준 적용
- 미처리 문의 우선 확인 지원
- 핵심 운영 지표 조회 진입점 제공

### 입력

- `store_id`
- 검색 조건
- 필터 조건
- 정렬 조건

### 출력

- 문의 목록 집합
- 각 문의의 요약 정보
  - 상태
  - 문의 유형
  - 고객명
  - 최근 메시지 시각
  - 마지막 응답 시각
  - 연결된 챗봇 전환 여부

### 규칙

- 기본 정렬은 `최근 메시지 시각 내림차순`을 우선 기본값으로 한다.
- 미처리 우선 정렬은 `RESOLVED`가 아닌 상태를 상위 배치하는 방식으로 해석한다.

### 예외 처리

- 조회 가능한 문의가 없는 경우 빈 상태 안내를 노출한다.
- 검색 결과가 없더라도 필터 조건은 유지한다.

## 2.2 inquiry-detail-module

### 책임

- 문의 단위의 전체 맥락 표시
- 상태 전환
- 응답 작성
- 내부 메모 관리
- 요약 및 자동 감지 결과 표시
- 연결된 챗봇 세션 원문 맥락 접근 제공

### 입력

- `inquiry_id`

### 출력

- `case_context_bundle`
- 상태 전환 가능 목록
- 응답/메모 작성 결과

### 구성 영역

1. 문의 헤더 영역
2. 고객 정보 영역
3. 주문/상품 연계 정보 영역
4. 문의 메시지 타임라인 영역
5. 연결된 챗봇 세션 참조 영역
6. 자동화 보조 정보 영역
7. 내부 메모 영역
8. 응답 작성 영역

### 규칙

- 내부 메모는 고객에게 노출되지 않는다.
- 고객/주문/상품 연계 데이터가 일부 없어도 문의 상세는 열려야 한다.
- 요약과 자동 감지 결과는 보조 정보이며 최종 응답 책임은 운영자에게 있다.
- 연결된 `chatbot_session`이 있으면 운영자는 전체 원문 맥락을 확인할 수 있어야 한다.

### 예외 처리

- 연계 주문이 없으면 “주문 정보 미연결” 상태를 표시한다.
- 요약 생성 실패 시 원문 메시지만 표시하고 실패 로그를 남긴다.
- 자동 감지 실패 시 감지 결과 영역은 숨기지 않고 “분석 불가” 상태를 표시한다.
- 연결된 챗봇 세션을 불러오지 못하면 “챗봇 기록 조회 실패” 상태를 표시하되 문의 열람은 유지한다.

## 2.3 internal-note-module

### 책임

- 운영자 간 공유용 메모 생성/조회

### 규칙

- 메모는 고객 UI/API에 포함되지 않는다.
- 메모 수정 이력 관리 여부는 **추가 정의 필요**하다.

## 2.4 response-composer-module

### 책임

- 직접 입력 응답 작성
- 프리셋 삽입
- FAQ/정책 기반 참고 정보 사용

### 입력

- 원문 메시지
- 프리셋 목록
- FAQ/정책 기반 참고 정보

### 출력

- 발송 대상 응답 메시지

### 규칙

- 금지 답변 규칙과 충돌하는 경우 발송 전 경고가 필요하다.
- MVP에서는 AI 추천 응답 생성 기능을 포함하지 않는다.

## 3. 데이터 흐름

```text
문의 목록 조회
 -> 문의 선택
 -> case_context_bundle 조회
 -> 요약/감지 결과 조회
 -> 연결된 chatbot_session 참조 조회(있을 경우)
 -> 운영자 응답 또는 프리셋/정책 기반 응답 사용
 -> 상태 변경 및 이력 저장
```

## 4. 화면 간 의존 관계

- 문의 목록은 문의 상태 모델과 운영 지표 정의에 의존한다.
- 문의 상세는 도메인 모델과 권한 모델에 의존한다.
- 응답 작성은 지식 자산과 정책 규칙에 의존한다.

## 5. 입력 검증 규칙

- 문의 유형은 허용된 enum만 사용해야 한다.
- 내부 메모 내용은 빈 문자열 저장을 허용하지 않는 것이 바람직하다.
- 응답 본문 최대 길이는 **추가 정의 필요**하다.

## 6. 운영자 관점 예외 시나리오

1. 문의는 존재하지만 고객 snapshot이 외부 연동 지연으로 오래 업데이트되지 않은 경우
2. 주문번호는 있으나 외부 연동 주문 조회가 실패한 경우
3. 동일 문의에 여러 운영자가 동시에 응답하는 경우
4. 상태 변경 직전 새 고객 메시지가 도착한 경우
5. 연결된 챗봇 세션은 있으나 원문 참조 로딩에 실패한 경우

위 시나리오의 충돌 해소 규칙은 **추가 정의 필요**하다.
