# 챗봇과 handoff

## 목적

챗봇은 고객이 관리자 응대로 넘어가기 전에 먼저 사용하는 self-service 진입점이다.

self-service만으로 해결되지 않으면 고객은 관리자 응대로 handoff를 요청할 수 있다.

## 주요 화면

### 챗봇 세션 화면

이 화면은 아래 요소를 포함해야 한다.

- 챗봇 대화 타임라인,
- 고객 입력 영역,
- 챗봇 응답 영역,
- handoff 액션,
- 세션 상태.

## 주요 챗봇 흐름

1. 고객이 스토어를 선택한 뒤 챗봇 탭을 연다.
2. 고객이 챗봇 세션을 시작하거나 이어서 연다.
3. 고객이 질문을 입력한다.
4. 챗봇이 스토어/상품 지식을 바탕으로 응답한다.
5. 해결되지 않으면 고객이 handoff를 요청한다.
6. 시스템이 문의를 생성하거나 기존 문의와 연결한다.
7. 연결된 챗봇 맥락이 운영자에게 보이게 된다.

챗봇 세션은 기본적으로 하나의 선택된 스토어 맥락 안에서 시작하는 것으로 본다.

## handoff 규칙

- handoff는 챗봇 흐름 안에서 제공된다.
- handoff는 관리자 응대로 이어지는 연결 경로를 만든다.
- 챗봇 대화 맥락은 운영자에게 계속 보여야 한다.
- 고객은 연결된 문의를 통해 후속 대화를 이어갈 수 있어야 한다.
- 하나의 `chatbot_session`은 최대 하나의 `inquiry`와 연결된다.
- 같은 세션에서 handoff를 다시 요청하면 새 문의를 계속 만드는 대신 기존 연결 문의를 반환하는 방식으로 처리한다.

## 문의와의 관계

챗봇 세션은 문의와 같은 개념이 아니다.

별도 흐름으로 존재하며, 필요할 때 문의로 이어질 수 있다.

## 단순화 메모

이 명세는 아래 항목을 완전하게 정의하지 않는다.

- 상세한 세션 만료 정책,
- 고급 idempotency 처리,
- 챗봇 실패 재시도 규칙,
- 내부 답변 생성 메커니즘.

중요한 요구사항은 챗봇이 실제 self-service 진입점이자 handoff 경로로 작동해야 한다는 점이다.
