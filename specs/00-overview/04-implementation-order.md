# 구현 우선순위와 개발 순서

## 목적

이 문서는 새로 정리한 `specs/` 문서를 실제 구현 순서로 연결하기 위한 작업 기준이다.

핵심 원칙은 아래와 같다.

- 화면이 빨리 보일 것
- 기능 흐름이 빨리 이어질 것
- 구현 난이도가 높은 운영 디테일보다는 데모 가능한 기능을 먼저 완성할 것
- 한 기능을 끝까지 연결한 뒤 다음 기능으로 넘어갈 것

## 전체 개발 원칙

1. 먼저 **로그인, 최소 데이터 구조, snapshot 기반**을 잡는다.
2. 그다음 **운영자 문의 흐름**을 완성한다.
3. 그다음 **고객 문의 흐름과 스토어 선택**을 완성한다.
4. 이후 **claim/link와 필요한 경우의 mock verification**을 붙인다.
5. 그 후 **챗봇과 handoff**를 붙인다.
6. 이후 **지식/요약/안전 보조**를 얹는다.
7. 마지막에 **검색/지표/커머스 맥락 고도화**를 정리한다.

즉, 프로젝트의 중심은 항상 아래 흐름이다.

고객 문의 생성 → 운영자 확인 및 응답 → 고객 상태 확인 → 필요 시 챗봇 handoff 연결

여기에 신규 가입 고객의 경우 기존 외부 주문을 자신의 계정과 연결하는 claim/link 흐름이 추가된다.

## 1단계: 최소 데이터 구조와 공통 뼈대

먼저 아래 구조를 만든다.

- `inquiry`
- `inquiry_message`
- `internal_note`
- `store`
- `local_user_account`
- `customer_snapshot`
- `order_snapshot`
- `product_snapshot`
- 기본 상태값(`OPEN`, `WAITING`, `IN_PROGRESS`, `ON_HOLD`, `RESOLVED`)

이 단계에서 필요한 것:

- 문의 생성 가능
- 문의 상세 조회 가능
- 메시지 저장 가능
- 상태 변경 가능

이 단계에서 함께 정리할 세부 내용:

- 고객과 운영자는 모두 `local_user_account`를 쓰되, `role` 값으로만 구분한다.
- `inquiry`는 이 프로젝트의 중심 케이스이므로, 이후 기능이 붙더라도 이 테이블을 기준으로 화면과 관계를 확장한다.
- `customer_snapshot`, `order_snapshot`, `product_snapshot`은 외부 원천 데이터를 그대로 복제하는 것이 아니라, 화면에서 참고할 최소 맥락만 저장하는 구조로 시작한다.
- 공통 레이아웃은 과하게 만들 필요 없고, 로그인 이후 상단 헤더와 기본 내비게이션 정도면 충분하다.

이 단계에서 구현하면 좋은 화면/행동 예시:

- 고객 또는 운영자가 로그인 후 자신의 시작 화면으로 이동한다.
- 테스트용 문의 데이터를 하나 만들고, 목록과 상세가 모두 열리는지 확인한다.
- 운영자가 상태를 바꾸면 상세 화면에서 즉시 반영된다.

이 단계에서 과하게 만들지 말아야 하는 것:

- 세밀한 권한 분기
- 읽음/안읽음 고도화
- 이벤트 로그 저장
- 외부 연동 재시도 로직

이 단계에서는 아직 챗봇, 요약, 지표, 복잡한 연동은 붙이지 않아도 된다.

## 2단계: 운영자 앱 핵심 화면

다음으로 `operator-app`의 핵심 화면을 만든다.

구현 대상:

- 문의 목록 화면
- 문의 상세 화면
- 운영자 응답 입력
- 내부 메모 입력
- 상태 변경 UI

이 단계에서 함께 정리할 세부 내용:

- 문의 목록은 우선 `OPEN`, `WAITING`, `IN_PROGRESS` 같은 진행 중 상태를 중심으로 보여주면 된다.
- 문의 상세에서는 고객 메시지, 운영자 메시지, 내부 메모가 서로 구분되어 보여야 한다.
- 상태 변경은 드롭다운이나 버튼 그룹처럼 단순한 UI로 먼저 구현한다.
- 운영자가 응답을 남긴 뒤 목록으로 돌아가면 마지막 응답 시각이나 최근 메시지 기준으로 갱신되는 흐름이 보이면 충분하다.

이 단계에서 구현하면 좋은 화면/행동 예시:

- 왼쪽에는 문의 목록, 오른쪽에는 선택한 문의 상세가 보이는 2단 레이아웃
- 문의 상세 상단에 상태, 문의 유형, 스토어명 같은 핵심 정보 표시
- 답변 입력창과 내부 메모 입력창을 분리하여 혼동을 줄이는 구성

이 단계의 완료 기준:

- 운영자가 문의를 열고,
- 대화 내용을 보고,
- 응답을 남기고,
- 메모를 추가하고,
- 상태를 변경할 수 있다.

이 단계가 끝나면 프로젝트는 이미 “운영자용 고객지원 도구”로 시연 가능해야 한다.

즉, 이 시점에서는 아직 고객 포털이 완전하지 않아도 운영자 입장에서 문의를 읽고 처리하는 핵심 시연이 가능해야 한다.

## 3단계: 고객 앱 핵심 화면

그다음 `customer-app`의 문의 흐름을 붙인다.

구현 대상:

- 고객 로그인 화면
- 운영자 로그인 페이지로 이동하는 보조 링크
- 고객 지원 진입 화면
- 내 스토어 목록 / 스토어 검색
- 고객 문의 목록 화면
- 고객 문의 상세 화면
- 고객 메시지 입력

이 단계에서 함께 정리할 세부 내용:

- 고객 로그인 화면은 기본 진입 화면으로 두고, 하단이나 보조 링크로 운영자 로그인 페이지 이동을 제공한다.
- 고객 지원 진입 화면에서는 `내 스토어`와 `스토어 검색`이 한 화면 안에서 자연스럽게 이어지면 좋다.
- 문의 생성은 처음부터 복잡한 양식을 만들기보다, 스토어 선택 → 문의 유형 선택 → 첫 메시지 입력 정도의 간단한 흐름으로 시작한다.
- 고객 문의 목록은 본인 문의만 보여준다는 점이 중요하며, 상태와 최근 업데이트 시각 정도만 먼저 보여도 충분하다.

이 단계에서 구현하면 좋은 화면/행동 예시:

- 문의가 하나도 없을 때는 “첫 문의를 작성해보세요” 같은 빈 상태 화면 제공
- 스토어 검색 결과에서 바로 문의 시작 버튼 제공
- 문의 상세에서 고객이 새 메시지를 입력하면 타임라인 마지막에 바로 붙는 흐름

이 단계의 완료 기준:

- 고객이 문의를 만들고,
- 스토어를 선택하거나 검색하고,
- 자신의 문의만 보고,
- 메시지를 추가하고,
- 상태를 확인할 수 있다.

이 단계가 끝나면 최소한의 end-to-end 문의 흐름이 완성된다.

즉, 고객이 문의를 만들고 운영자가 보고 응답하고 다시 고객이 상태를 보는 기본 루프가 연결되어야 한다.

## 4단계: 계정 연결과 주문 claim

이제 외부 snapshot과 로컬 계정을 연결하는 흐름을 붙인다.

구현 대상:

- 주문 claim / link 화면
- claim 후보 조회
- `customer_account_link` 반영
- `customer_store_link` 반영
- 필요한 경우의 mock verification provider 시뮬레이션

이 단계에서 함께 정리할 세부 내용:

- 고객이 막 가입한 경우에도 외부 주문을 자기 계정과 연결할 수 있어야 한다.
- 기본 흐름은 주문번호, 이름, 전화번호 힌트, 이메일 등 낮은 민감도 정보 기반 후보 조회로 시작한다.
- 더 강한 확인이 필요할 때만 mock verification을 보조 단계로 사용한다.
- claim 성공 후에는 `customer_account_link`, `customer_store_link`가 반영되어 고객 화면의 `내 스토어`가 자연스럽게 늘어나야 한다.

이 단계에서 구현하면 좋은 화면/행동 예시:

- “내 주문 찾기” 화면에서 주문번호 입력 후 후보 목록 확인
- claim 진행 중 / 성공 / 실패 상태를 분리해 보여주는 단순 상태 UI
- mock verification 코드 입력 화면을 별도 팝업 또는 한 단계 화면으로 제공

이 단계의 완료 기준:

- 필요하면 기존 주문을 claim/link 할 수 있고,
- claim 전에는 주문 연결이 미확정 상태로 남을 수 있고,
- claim 성공 후에는 외부 customer/order snapshot과 로컬 계정이 연결된다.

mock verification은 이 단계의 기본 경로가 아니라 보조 fallback으로만 사용한다.

핵심은 “외부 주문과 로컬 계정은 처음부터 자동으로 같지 않다”는 구조를 사용자 흐름으로 보여주는 것이다.

## 5단계: 대화와 첨부 기능

이제 대화 경험을 보강한다.

구현 대상:

- 메시지 타임라인 정리
- 발신자 구분 표시
- 이미지 첨부 업로드
- 첨부 미리보기
- `content_raw` / `content_display` 구조 반영

이 단계에서 함께 정리할 세부 내용:

- 고객 메시지와 운영자 메시지는 말풍선, 정렬, 배지 등으로 명확히 구분되어야 한다.
- 메시지에는 최소한 작성 시각이 보여야 하고, 긴 대화에서도 읽기 어렵지 않게 정렬되어야 한다.
- 이미지 첨부는 여러 파일 고도화보다 먼저 “한 장 이상 업로드 가능하고 미리보기 가능” 수준이면 충분하다.
- `content_raw`는 원문 저장, `content_display`는 화면 표시용 텍스트라는 역할 차이를 문서와 구현에서 함께 유지한다.

이 단계에서 구현하면 좋은 화면/행동 예시:

- 이미지 썸네일 클릭 시 확대 보기
- 상태 변경 같은 주요 변화는 간단한 시스템 메시지처럼 타임라인에 보여주기
- 모바일/좁은 화면에서도 메시지 영역이 깨지지 않는 기본 레이아웃 유지

이 단계의 완료 기준:

- inquiry 대화가 메시지 중심 UI로 보이고,
- 이미지 첨부가 기능적으로 작동한다.

## 6단계: 챗봇과 handoff

이제 챗봇 기능을 붙인다.

구현 대상:

- 챗봇 세션 시작
- 챗봇 질문/응답
- 고객 화면의 `문의` / `챗봇` 탭 구조
- handoff 버튼
- handoff 시 `inquiry` 생성 또는 기존 연결 문의 반환

이 단계에서 함께 정리할 세부 내용:

- 챗봇은 고객이 먼저 self-service를 시도하는 흐름으로 배치한다.
- FAQ나 정책 문서를 기반으로 답할 수 있는 단순 질문부터 처리하게 두면 된다.
- handoff가 발생하면 운영자가 챗봇 대화 맥락을 볼 수 있어야 하므로, 연결된 `chatbot_session`과 `inquiry` 관계가 분명해야 한다.
- 고객 화면에서는 문의 탭과 챗봇 탭이 섞이지 않도록 진입 지점과 현재 상태를 분명히 보여주는 것이 중요하다.

이 단계에서 구현하면 좋은 화면/행동 예시:

- “관리자에게 넘기기” 버튼
- handoff 직후 생성된 문의 상세로 이동하거나, 이미 연결된 문의가 있으면 그 문의로 이동
- 운영자 화면에서 챗봇 대화 요약 또는 최근 몇 개 메시지를 함께 보는 구성

이 단계의 완료 기준:

- 고객이 챗봇을 먼저 사용하고,
- 해결되지 않으면 handoff를 요청하고,
- 운영자가 연결된 챗봇 맥락을 볼 수 있다.

이 단계가 끝나면 제품의 핵심 주제가 거의 완성된다.

즉, 고객 self-service와 관리자 응대 연결이라는 이 프로젝트의 중요한 특징이 실제 데모 흐름으로 보여야 한다.

## 7단계: 지식 자산과 운영 보조

다음으로 운영자 효율을 높이는 기능을 붙인다.

구현 대상:

- FAQ 조회
- 정책 문서 조회
- 응답 프리셋 삽입
- 챗봇 지식 파일 업로드/목록

이 단계에서 함께 정리할 세부 내용:

- 운영자는 문의 답변을 작성할 때 FAQ, 정책, 응답 프리셋을 참고할 수 있어야 한다.
- 챗봇 지식 파일은 먼저 업로드/목록/활성 여부 정도만 관리하면 충분하다.
- 지식 자산은 운영자 전용 관리 화면과 챗봇 응답 보조에 동시에 연결되는 구조로 이해하면 된다.
- 고급 버전 관리나 승인 워크플로는 이 단계에서 넣지 않는다.

이 단계에서 구현하면 좋은 화면/행동 예시:

- 답변 입력창 옆에 프리셋 삽입 버튼 제공
- FAQ/정책 문서를 제목 기준 목록으로 보여주고, 클릭 시 본문 열람
- 챗봇 지식 파일 목록에서 활성 여부만 간단히 토글

이 단계의 완료 기준:

- 운영자가 답변에 참고할 자료를 볼 수 있고,
- 챗봇이 사용할 기본 지식 파일을 관리할 수 있다.

## 8단계: 요약과 안전 보조

이제 AI 보조 기능을 붙인다.

구현 대상:

- 문의 요약
- 챗봇 세션 요약
- 악성 표현 감지
- 고객 노출용 표시 완화

이 단계에서 함께 정리할 세부 내용:

- 요약은 긴 대화를 빠르게 이해하기 위한 보조 기능으로 둔다.
- 악성 표현 감지는 운영자의 판단을 대신하는 것이 아니라, 주의를 주는 보조 신호로만 사용한다.
- 표시 완화는 실제 원문을 지우는 것이 아니라 화면 노출을 조절하는 방식으로 단순하게 시작하면 된다.
- 운영자가 “요약 보기” 또는 “안전 경고 확인”을 쉽게 찾을 수 있도록 위치를 명확히 한다.

이 단계에서 구현하면 좋은 화면/행동 예시:

- 문의 상세 우측 패널에 요약 카드 표시
- 특정 메시지에 경고 배지 표시
- 민감하거나 과격한 표현 일부를 `[가림 처리]`처럼 보여주는 단순 표시 규칙

이 단계의 완료 기준:

- 운영자가 요약을 참고할 수 있고,
- 시스템이 고객 노출 메시지에 대해 표시 완화 여부를 판단하거나, 운영자에게 주의 신호를 줄 수 있다.

이 단계에서는 AI를 운영자 대체가 아니라 보조 기능으로 유지해야 한다.

이 기준이 무너지면 프로젝트가 다시 과한 자동화 시스템처럼 보일 수 있으므로, 항상 보조 역할에 머물게 해야 한다.

## 9단계: 검색과 지표

이제 운영자 생산성을 위한 조회 기능을 붙인다.

구현 대상:

- 고객명/주문번호/상품명/상태/유형/기간 검색
- 문의 수
- 평균 첫 응답 시간
- 해결 완료율

이 단계에서 함께 정리할 세부 내용:

- 검색은 먼저 운영자가 실제로 자주 찾을 조건 위주로 단순하게 붙인다.
- 지표는 실시간 완벽 집계보다, 현재 DB에서 계산 가능한 기본 KPI를 보여주는 수준이면 충분하다.
- 별도 검색 전용 테이블이나 집계 배치 없이도 시작할 수 있다는 점을 유지한다.
- 필터와 KPI는 운영자 홈이나 문의 목록 상단에 자연스럽게 배치하면 된다.

이 단계에서 구현하면 좋은 화면/행동 예시:

- 상태, 기간, 스토어를 고를 수 있는 필터 바
- 상단 KPI 카드 3~4개
- 현재 필터 결과에 맞는 문의 목록 즉시 갱신

이 단계의 완료 기준:

- 운영자가 문의를 빠르게 찾을 수 있고,
- 기본 운영 상태를 KPI로 확인할 수 있다.

## 10단계: 커머스 맥락 고도화

마지막으로 외부 커머스 맥락을 붙인다.

구현 대상:

- customer/order/product snapshot 조회
- 문의 상세에서 연결 맥락 표시
- 수동 새로고침 액션

이 단계에서 함께 정리할 세부 내용:

- 운영자가 문의를 읽을 때 고객, 주문, 상품 맥락을 한눈에 볼 수 있어야 한다.
- snapshot은 외부 원천의 모든 정보를 복제하기보다, 문의 처리에 필요한 최소 정보만 보여주면 된다.
- 수동 새로고침은 복잡한 동기화 파이프라인 대신 버튼 한 번으로 최신 snapshot을 다시 불러오는 정도면 충분하다.
- 이 단계는 앞선 핵심 흐름이 완성된 뒤 완성도를 높이는 작업으로 이해해야 한다.

이 단계에서 구현하면 좋은 화면/행동 예시:

- 문의 상세 우측 사이드 패널에 고객/주문/상품 카드 배치
- 주문 상태, 상품명, 주문일 같은 핵심 필드만 먼저 노출
- “최신 정보 다시 불러오기” 버튼 제공

이 단계의 완료 기준:

- 운영자가 문의를 볼 때 고객/주문/상품 정보를 함께 이해할 수 있다.

이 단계는 제품 완성도를 높이지만, 문의 흐름보다 먼저 구현할 필요는 없다.

## 추천 실제 작업 순서

가장 추천하는 실제 구현 순서는 아래와 같다.

1. DB 초안 + 최소 snapshot 구조
2. 고객/운영자 로그인 분기
3. 운영자 문의 목록/상세
4. 운영자 메시지/메모/상태 변경
5. 고객 문의 생성/조회/메시지
6. 스토어 선택/검색
7. 주문 claim/link
8. 필요한 경우의 mock verification
9. 첨부 기능
10. 챗봇 세션 + handoff
11. FAQ/정책/프리셋/지식 파일
12. 요약/악성 표현 감지
13. 검색/지표
14. customer/order/product snapshot 맥락 고도화

## 개발 중 절대 잊지 말 것

- 먼저 보이는 기능을 만들고, 그다음 정교하게 다듬는다.
- 운영 디테일 때문에 핵심 흐름 구현이 밀리면 안 된다.
- “완벽한 구조”보다 “끝까지 연결된 기능 흐름”이 우선이다.
- 이 프로젝트의 핵심은 많은 규칙이 아니라, 고객과 운영자가 실제로 사용할 수 있는 화면과 흐름이다.
