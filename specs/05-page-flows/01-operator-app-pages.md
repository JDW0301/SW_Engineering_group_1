# 운영자 앱 페이지

## 목적

이 문서는 운영자 앱에서 실제로 필요한 페이지를 단순한 사용자 흐름 기준으로 정리한다.

운영자 앱의 핵심은 많은 화면을 두는 것이 아니라, `문의 목록`과 `문의 상세` 두 화면 안에서 대부분의 작업을 끝낼 수 있게 만드는 것이다.

## 운영자 앱 페이지 목록

운영자 앱의 기본 페이지는 아래 세 가지로 본다.

1. 문의 목록 페이지
2. 문의 상세 페이지
3. 지식/프리셋 관리 페이지

스냅샷 확인과 수동 재동기화는 독립 운영 콘솔보다 `문의 상세 페이지` 안의 보조 기능으로 우선 다룬다.

## 1. 문의 목록 페이지

### 목적

운영자가 지금 어떤 문의를 먼저 처리해야 하는지 빠르게 판단하는 triage 화면이다.

### 포함 기능

- 고객명 검색
- 주문번호 검색
- 상품명 검색
- 문의 상태 필터
- 문의 유형 필터
- 기간 preset 선택
  - 오늘
  - 최근 7일
  - 최근 30일
- KPI 요약 표시
  - 문의 수
  - 평균 첫 응답 시간
  - 해결 완료율
- 문의 목록 표시
- degraded indicator 표시
- linked chatbot session indicator 표시

### 목록 각 행에 보여야 하는 것

- inquiry id 또는 간단 식별자
- 상태 배지
- 문의 유형
- 고객 이름
- 주문/상품 요약
- 최근 메시지 시각
- 최근 응답 시각
- 챗봇 연계 여부
- 스냅샷 누락 또는 최신화 필요 여부

### 주요 흐름

1. 운영자가 페이지에 진입한다.
2. 기본 필터 상태로 문의 목록과 KPI를 본다.
3. 검색과 필터를 조정한다.
4. 우선 처리할 문의를 선택한다.
5. 문의 상세 페이지로 이동한다.

### 단순화 메모

- 검색과 KPI는 별도 페이지가 아니라 이 페이지 안의 영역이다.
- 커스텀 날짜 범위나 복잡한 검색 DSL은 MVP 범위 밖으로 둔다.

## 2. 문의 상세 페이지

### 목적

운영자가 실제 상담 처리를 수행하는 핵심 작업 화면이다.

### 포함 기능

- 문의 헤더
  - inquiry id
  - 현재 상태
  - 문의 유형
  - 생성 시각
  - 담당자
- 고객/주문/상품 요약 영역
- 문의 메시지 타임라인
- 운영자 메시지 입력 영역
- 첨부 업로드 및 미리보기
- 내부 메모 패널
- 상태 변경 액션
- 재오픈 액션
- 담당자 변경 액션
- linked chatbot context 패널
- 요약/안전/지식 보조 패널
- degraded context 안내
- 필요 시 manual resync 액션

### 이 페이지에서 보조적으로 보여야 하는 정보

- 최신 inquiry summary
- abuse detection 결과
- 관련 FAQ / 정책 / 프리셋
- 연결된 chatbot session summary 또는 원문 접근 링크

### 주요 흐름

1. 운영자가 목록에서 문의를 선택한다.
2. 고객, 주문, 상품, 문의 대화, 챗봇 연결 맥락을 확인한다.
3. 필요하면 요약과 안전 보조 정보를 참고한다.
4. 운영자가 응답을 작성한다.
5. 내부 메모를 남기거나 상태를 변경한다.
6. 완료 후 목록으로 돌아가 다음 문의를 처리한다.

### 상태 처리 원칙

최소 상태 흐름은 아래 정도면 충분하다.

- `OPEN`
- `WAITING`
- `IN_PROGRESS`
- `ON_HOLD`
- `RESOLVED`

재오픈은 별도 복잡한 워크플로가 아니라, 완료된 문의를 다시 `OPEN`으로 돌리는 단순 액션으로 본다.

### 단순화 메모

- 동시 수정 충돌 정책은 깊게 다루지 않는다.
- linked chatbot context는 별도 대형 화면보다 보조 패널 우선으로 둔다.
- snapshot 누락은 문의 처리 자체를 막지 않는다.

## 3. 지식/프리셋 관리 페이지

### 목적

운영자가 챗봇 지식 파일과 답변 참고 자산을 관리하는 단순 운영 페이지다.

### 포함 기능

- `.txt` 지식 파일 다중 업로드
- 현재 active 파일 목록 표시
- 파일별 업로드 성공/실패 결과 표시
- 최신 업로드 우선 규칙 안내
- 관련 FAQ / 정책 / 프리셋 목록의 단순 조회 또는 관리 진입

### 주요 흐름

1. 운영자가 지식 관리 페이지를 연다.
2. 현재 active 파일 상태를 확인한다.
3. 새 파일을 업로드한다.
4. 성공/실패 결과를 확인한다.

### 단순화 메모

- 이 페이지는 운영 편의를 위한 관리 화면이다.
- 요약과 안전 감지 결과 자체는 여기보다 문의 상세 페이지 안에서 보는 것이 중심이다.

## 스냅샷 맥락 처리 원칙

customer/order/product snapshot 정보는 운영자 문의 상세 안의 보조 맥락으로 다루는 것이 기본이다.

문의 처리에 필요한 최소 동작은 아래 정도면 충분하다.

- linked snapshot 요약 표시
- `미연결`, `최신화 필요` 같은 간단한 degraded 상태 표시
- 필요 시 entity 단위 manual resync 액션 제공

이 프로젝트는 별도의 복구 콘솔이나 이벤트 로그 화면을 핵심 페이지 집합에 포함하지 않는다.

## 운영자 앱 전체 흐름 요약

1. 문의 목록에서 대상을 찾는다.
2. 문의 상세에서 맥락을 확인하고 처리한다.
3. 필요하면 지식 관리 페이지를 보조적으로 연다.
4. 다시 문의 상세 또는 목록으로 돌아와 운영 흐름을 이어간다.
