# 선행 작성 우선순위

## 1. 목적

본 문서는 상세 명세를 어떤 순서로 작성해야 서로 충돌하지 않는지 정리하기 위한 우선순위 문서이다.

## 2. 우선순위 목록

## 1순위. 제품 범위 및 MVP 경계

### 왜 선행되어야 하는가

- 어떤 기능을 이번 단계에서 확정할지 정해야 다른 문서의 범위가 불필요하게 커지지 않는다.
- 웹/앱 범위, 소비자 화면 포함 여부, 외부 연동 포함 여부가 이후 설계 깊이를 결정한다.

### 이 문서가 기준이 되는 대상

- 도메인 모델 범위
- 권한 모델 범위
- 화면 수와 모듈 분리 범위
- AI 기능 포함 범위

## 2순위. 도메인 모델 및 데이터 사전

### 왜 선행되어야 하는가

- 명명 체계의 기반이 된다.
- API 입력/출력, 검색 조건, 상태 모델, 통계 계산, AI 입력 구조를 모두 정의할 수 있게 된다.

### 이 문서가 기준이 되는 대상

- 통합 상담 화면
- 문의/채팅/첨부 파일 데이터 구조
- 외부 연동 포맷
- 보관 정책 단위

## 3순위. 역할 및 권한 모델

### 왜 선행되어야 하는가

- 어떤 사용자가 어떤 정보와 기능에 접근할 수 있는지 먼저 정해야 보안 정책과 화면 정책이 안정된다.
- 개인정보 마스킹과 내부 메모 노출 여부도 권한 모델에 의존한다.

### 이 문서가 기준이 되는 대상

- 인증 흐름
- 화면별 접근 제어
- 상태 전환 권한
- 정책 문서 수정 권한

## 4순위. 문의 상태 모델 및 전환 규칙

### 왜 선행되어야 하는가

- 문의 목록, 정렬, SLA, 통계, AI handoff, 알림 규칙의 기준 상태가 된다.
- 상태 모델이 없으면 응답 시간과 해결 완료율도 정확히 정의할 수 없다.

### 이 문서가 기준이 되는 대상

- 문의 운영 화면
- 채팅 연결 규칙
- 상담 메모 처리
- 운영 지표 계산

## 5순위. 보안·개인정보·보관 정책

### 왜 선행되어야 하는가

- 저장 가능한 데이터, 마스킹 범위, 파일 보관, AI 처리 가능 범위를 제한한다.
- 구현 이후 바꾸기 어려운 정책이므로 초기에 고정하는 것이 유리하다.

### 이 문서가 기준이 되는 대상

- 데이터 저장 구조
- 첨부 파일 저장 정책
- 접근 로그
- AI 입력 제한

## 6순위. 예외 처리 및 공통 제약

### 왜 선행되어야 하는가

- 정상 흐름만 있는 명세는 실제 구현에 바로 도움이 되지 않는다.
- 예외 상황이 먼저 정의되어야 API와 UI가 일관된 오류 처리를 할 수 있다.

### 이 문서가 기준이 되는 대상

- 입력 검증
- 권한 오류 처리
- 업로드 실패 처리
- AI 실패 fallback
- 연동 실패 재시도 정책

## 7순위. 통합 상담 작업공간 및 문의 운영 상세

### 왜 선행되어야 하는가

- 실제 사용자 가치가 가장 먼저 드러나는 핵심 화면이기 때문이다.
- 위 1~6순위 문서를 입력값으로 사용해 상세 플로우를 구체화할 수 있다.

## 8순위. 채팅 및 첨부 파일

### 왜 선행되어야 하는가

- 문의 상태 모델, 권한 모델, 보관 정책을 기반으로 설계해야 한다.

## 9순위. AI 보조 기능 및 지식 자산

### 왜 선행되어야 하는가

- 기본 워크플로우가 먼저 안정되어야 AI가 어디에 개입하는지 정의할 수 있다.
- 지식 자산 모델과 정책 우선순위를 따로 정해야 한다.

## 10순위. 검색, 지표, 외부 연동

### 왜 후행되는가

- 검색은 데이터 구조와 명명 체계에 의존한다.
- 지표는 상태 모델과 시간 기준에 의존한다.
- 외부 연동은 내부 도메인 모델이 확정된 후 맞추는 것이 안전하다.

## 3. 의존 관계 요약

```text
제품 범위
  -> 도메인 모델
    -> 권한 모델
    -> 문의 상태 모델
      -> 보안/보관 정책
      -> 예외 처리 규칙
        -> 통합 상담 작업공간
        -> 채팅/첨부 파일
        -> AI 보조 기능
        -> 검색/지표
        -> 외부 연동
```

## 4. 현재 문서 세트 적용 원칙

- `foundation/` 문서가 `modules/`와 `platform/` 문서보다 먼저 읽히고 먼저 갱신되어야 한다.
- `modules/` 문서는 반드시 `foundation/` 문서의 용어와 상태값을 그대로 따른다.
- `platform/` 문서는 예외 처리와 공통 제약을 정의하며, 개별 기능 문서보다 우선 적용된다.
