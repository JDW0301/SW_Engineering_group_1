# 커머스 데이터 동기화

## 목적

운영자는 지원 케이스를 처리할 때 고객, 주문, 상품 맥락을 함께 볼 수 있어야 한다.

이 기능은 외부 커머스 원천에서 그 맥락을 가져온다.

## 핵심 아이디어

이 프로젝트는 지원 데이터베이스를 외부 커머스 시스템의 전체 복제본으로 취급하지 않는다.

대신 지원 작업에 필요한 최소한의 내부 snapshot 또는 reference만 유지한다.

즉 customer, order, product 정보는 이 제품의 핵심 lifecycle을 직접 소유하는 데이터가 아니라, 외부 원천에서 가져와 문의 처리를 돕는 지원 맥락으로 본다.

## 주요 맥락 유형

### 고객 맥락

고객 식별과 기본 고객 관련 지원 이력을 파악하기 위해 사용한다.

### 주문 맥락

주문 관련 이슈를 이해하기 위해 사용한다.

### 상품 맥락

상품 관련 이슈를 이해하기 위해 사용한다.

## 운영자 사용 방식

운영자는 아래 동작을 할 수 있어야 한다.

- 문의 상세에서 연결된 고객/주문/상품 요약을 본다.
- 일부 연결 데이터가 없거나 오래되었어도 문의 처리를 계속한다.
- 필요하면 수동 새로고침을 요청한다.

## 동기화 모델

간소화된 모델은 아래와 같다.

- 외부 시스템이 원본이다.
- 프로젝트는 지원용 맥락을 유지한다.
- webhook 기반 업데이트를 지원한다.
- 운영자는 수동 새로고침을 요청할 수 있다.

## 외부 identity와 로컬 계정 연결

이 프로젝트는 외부 스토어의 customer/order 데이터를 가져올 때, 외부 원본 식별자와 로컬 로그인 계정을 같은 것으로 취급하지 않는다.

대신 아래 원칙을 사용한다.

- 외부 customer/order/product는 snapshot으로 반영한다.
- 로컬 사용자는 우리 서비스에서 별도로 로그인한다.
- 필요할 때 로컬 계정이 외부 customer 또는 order를 claim/link 한다.
- 외부 식별자는 snapshot 쪽의 기준값으로 유지하고, 로컬 계정 연결은 별도 링크 기록으로 남긴다.

claim의 최종 목적은 개별 `order_claim_request`를 영구 소유 데이터로 삼는 것이 아니라, 적절한 `customer_account_link`와 `customer_store_link`를 만드는 데 있다.

주문 연결 전에는 문의가 스토어 기준으로 먼저 생성될 수 있고, 성공한 claim 이후에만 `order_snapshot` 연결을 확정하는 흐름을 기본으로 본다.

## mock verification provider

학교 프로젝트 범위에서는 실제 통신사 또는 본인확인 기관 연동을 구현하지 않는다.

다만 구조적으로는 외부 verification provider가 존재한다고 가정하고, 아래 수준으로 시뮬레이션할 수 있다.

- 고객이 기본 정보를 입력한다.
- 외부 verification provider에 확인을 요청하는 흐름을 호출한다.
- 프로젝트에서는 mock verification 결과를 반환한다.
- 이 결과를 주문 claim/link 흐름의 보조 확인 수단으로만 사용한다.

이 mock verification은 실제 법적 본인확인을 의미하지 않으며, 주민등록번호 같은 민감 식별자 수집을 전제하지 않는다.

또한 이 기능은 애매한 claim 상황에만 사용하는 fallback으로 보고, 기본 계정 연결 모델의 중심에는 두지 않는다.

## 단순화 메모

이 명세는 아래를 완전하게 정의하지 않는다.

- 이벤트 로그 화면,
- 처리 추적 UI,
- 재시도/백오프 전략,
- payload 번역 디테일,
- 고급 snapshot 복구 워크플로.

핵심 요구사항은 완전한 연동 관측성이 아니라 기능적으로 쓸 수 있는 운영자 맥락이다.
