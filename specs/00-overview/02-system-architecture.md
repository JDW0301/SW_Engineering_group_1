# 시스템 구조

## 전체 구조

이 프로젝트는 공유 백엔드와 공유 데이터베이스를 사용하는 하나의 웹 제품으로 구현한다.

시스템 구조는 modular monolith를 기준으로 한다.

## 앱 엔트리

사용자 진입은 로그인 단계에서 먼저 분기된다.

- 기본 로그인 화면은 고객 로그인 기준으로 시작한다.
- 화면 하단 또는 별도 액션을 통해 운영자 로그인 페이지로 이동할 수 있다.
- 고객 로그인 성공 시 `customer-app`으로 진입한다.
- 운영자 로그인 성공 시 `operator-app`으로 진입한다.

이 구조는 하나의 제품 안에서 두 사용자 표면을 분리하는 가장 단순한 방식으로 본다.

### `customer-app`

주요 책임:

- 문의 생성,
- 문의 상세 및 메시지 흐름,
- self-service 챗봇,
- 챗봇에서 관리자 응대로의 handoff,
- 고객이 볼 수 있는 상태 확인.

### `operator-app`

주요 책임:

- 문의 작업공간,
- 맥락이 풍부한 문의 상세,
- 내부 메모,
- 상태 관리,
- 지식 지원,
- 검색과 지표,
- 커머스 맥락 새로고침.

## 인증과 계정 연결 구조

이 제품은 두 종류의 식별 축을 가진다.

1. 로컬 로그인 계정
2. 외부 스토어에서 들어오는 customer/order/product snapshot 식별자

이 둘은 처음부터 같은 것으로 취급하지 않는다.

대신 아래 원칙을 사용한다.

- 외부 스토어 데이터는 snapshot으로 가져온다.
- 로컬 사용자는 로그인 후 우리 서비스의 계정을 가진다.
- 주문 또는 고객 정보가 필요할 때 로컬 계정과 외부 snapshot을 claim/link 흐름으로 연결한다.

강한 확인 절차가 필요한 경우에만 학교 프로젝트용 mock verification provider를 보조 수단으로 사용할 수 있다.

이 문서에서는 구조만 설명하고, claim/link와 verification의 세부 동작은 관련 기능 문서에서 정의한다.

## 공유 핵심 모듈

두 앱은 아래 논리 모듈을 공유한다.

1. 문의 관리 모듈
2. 고객 포털 모듈
3. 대화 및 첨부 모듈
4. 안전 보조 및 지식 지원 모듈
5. 검색 및 지표 모듈
6. 커머스 맥락 연동 모듈

## 구조 원칙

기능 모듈은 책임 기준으로 나누되, 전체 시스템은 학생 팀이 구현할 수 있을 정도로 단순해야 한다.

즉 아래를 의미한다.

- 배포 가능한 제품은 하나,
- 영속성 계층은 공유,
- 기능 경계는 명확히 유지,
- 마이크로서비스 분리는 하지 않음,
- 핵심 데모를 위해 복잡한 이벤트 기반 인프라에 의존하지 않음.

## 단순화 규칙

설계 선택이 애매할 때는 아래를 우선한다.

- 정상 동작 중심 흐름,
- 실제로 보이는 화면,
- 단순한 데이터 흐름,
- 공유 백엔드 모듈.

엔터프라이즈처럼 보이기 위해 복잡도를 추가하지 않는다.
